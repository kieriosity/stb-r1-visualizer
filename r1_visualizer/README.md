# STB R-1 Visualizer

A form-faithful, client-side viewer for STB Form R-1 canonical JSON. The **same
bundle** runs standalone (for local testing) and embeds into WordPress via a
shortcode — no Python or PHP rendering server in production. The static JSON
*is* the data source.

## Architecture

The viewer renders a **near-pixel facsimile of the printed Form R-1**. Rather
than building each schedule's table by hand, it reconstructs the official xlsx
form template (`forms/R1-7-31-2026.xlsx`) cell-for-cell and overlays each
submission's numeric values into the blank value cells.

```
forms/R1-7-31-2026.xlsx ──gen-form-template.py──> src/formTemplate.json
                                                    (grid: cells, borders,
                                                     widths, fonts, spans)

manifest.json + STB-R1-*.json  ──fetch──>  Preact bundle (r1-viewer.js/.css)
        (static assets)                     ├─ dataSource.js     swappable backend
                                            ├─ formTemplate.json bundled form grid
                                            └─ FormFacsimile.jsx renders grid +
                                                                 overlays values
```

- **`FormFacsimile.jsx`** draws every form page (front matter, instruction
  pages, and all data schedules) from the template grid, then fills value cells:
  - rows are matched to data records by **line number** (then account number);
  - value columns are matched to data keys by **header-text tokens**, computed
    **per header band**, so sub-tables that reuse the same physical columns for
    different values (e.g. schedule 450) map correctly;
  - static blocked-cell markers (`N/A`, `XXXXXX`) are preserved from the
    template, while fillable placeholders like `(   )` can still receive data.
- **`formTemplate.json`** is generated by `scripts/gen-form-template.py`
  (Python + openpyxl). Re-run it if the official form template changes.
- **`dataSource.js`** is an interface. Today it static-fetches JSON. A
  FastAPI-backed impl (see `../r1_api`) can drop in for server-side
  search/filtering without touching the renderer.
- The earlier per-shape renderers (`schedules/*`, `ScheduleView.jsx`) are kept
  for reference but are no longer used by the facsimile view.

### Regenerating the form template

```bash
# WSL, stb-r1 conda env (needs openpyxl):
python scripts/gen-form-template.py     # -> r1_visualizer/src/formTemplate.json
```

## Local testing (standalone)

```bash
npm install
npm run dev          # http://localhost:5173
```

A Vite dev plugin serves the sibling `../stb_r1_json/` folder at `/data/*` and
synthesizes `/data/manifest.json` — nothing is copied. Use the carrier / year /
version pickers to browse all 287 submissions.

## Production build

```bash
npm run build        # -> dist/r1-viewer.js, dist/r1-viewer.css
```

The bundle exposes `window.mountR1Viewer(el, opts)` and auto-mounts any
`[data-r1-viewer]` element. It inlines the form template (`formTemplate.json`),
so the JS is ~1.1 MB raw / ~140 KB gzipped. (To trim initial load it can be
switched to a lazy `import('./formTemplate.json')` code-split chunk.)

A `?sched=<id>` query param (or `#<id>` hash) deep-links to a specific schedule
or page, e.g. `?sched=410` or `?sched=Cover`.

## WordPress

1. `npm run build`, then copy `dist/r1-viewer.js` and `dist/r1-viewer.css` into
   `wordpress/assets/`.
2. Copy the `wordpress/` folder into `wp-content/plugins/r1-visualizer/` and
   activate the plugin.
3. Generate the manifest and upload the data:
   ```bash
   node scripts/gen-manifest.mjs ../stb_r1_json
   # upload ../stb_r1_json/*.json + manifest.json -> wp-content/uploads/r1-data/
   ```
   Adjust `R1_DATA_BASE` in `wordpress/r1-visualizer.php` if you host data
   elsewhere (e.g. a CDN).
4. Drop the shortcode into any page:
   ```
   [r1_viewer carrier="BNSF" year="2025"]
   ```
   Omit `carrier`/`year` to default to the latest submission with full pickers.

Styles are scoped under `.r1-app` to avoid clashing with the host theme.

## Refining form fidelity

Static labels, column headers, instruction text, borders, column widths and
fonts all come straight from the official template, so they match the printed
form automatically. The only logic is mapping each submission's values into the
right cells — see the per-band header-token matching in `FormFacsimile.jsx`. If
a new schedule maps values incorrectly, check (a) its `columnSpec.json` column
keys and (b) that its header labels tokenise to those keys.
